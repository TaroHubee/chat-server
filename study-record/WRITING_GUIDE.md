# study-record 執筆スタイルガイド（Copilot 指示書）

このファイルは Copilot が study-record 内の記事を作成・更新するときの指示書です。

---

## シリーズ概要

**タイトル**: 「Rustでチャットサーバーを作る」  
**対象読者**: Rust初心者（チュートリアルを少しやった程度）  
**目的**: 実装しながらRustの概念を学んだ記録をQiitaに投稿する

---

## ファイル命名規則

```
p{Phase番号}_{連番:02d}_{内容}.md
例: p1_01_ownership_borrowing.md
    p2_01_threads_arc_mutex.md
```

---

## 記事の構成ルール

### 必須要素（この順番で）

1. **タイトル（H1）**: `# Rustでチャットサーバーを作る #{通し番号} — {内容}`
2. **導入**: 何の話をするか・なぜ必要かを2〜3文で
3. **本文**: H2 で章立て。概念 → コード → 解説の順
4. **まとめ（H2）**: 箇条書きで要点を整理
5. **次の記事への繋ぎ**: 最後の1文で次の記事に誘導

### 通し番号の管理

全記事を通した連番（Phase をまたいでも継続）。  
現在の最終番号は `p2_02` = **#6**。次は #7 から。

---

## 文体・トーン

### 視点・語り手

**記事は「Copilotに教わりながら学習した私（筆者）」が書いたものとして書く。**

- Copilot が説明するのではなく、**学習者である私が「わかった！」「ここで詰まった」と振り返る一人称の語り口**にする
- 「〜と教わりました」「〜でつまずきました」「〜がやっとわかりました」など、学習体験を語る表現を使う
- 上から目線の解説文ではなく、**同じく初心者だった読者に向けて「こう理解した」と共有するトーン**

例（NG）:
> Rustの所有権は〜という仕組みです。

例（OK）:
> 最初は「所有権って何？」状態でしたが、「変数が値を所有する」というイメージで一気に理解できました。

### その他

- **です・ます調**（敬体）
- 難しい概念は**例え話**を使って説明（例: Mutex = トイレの鍵）
- 「なぜそれが必要か」を先に書いてからコードを示す
- 他言語（TypeScript・Java・Python）との比較を適宜入れる

---

## コードブロックのルール

- 言語指定を必ず書く（` ```rust `・` ```bash ` など）
- コードにはインラインコメントで学習ポイントを補足する
- コンパイルエラーの例を示すときは `// ← エラー！` などでマーク
- 実行コマンドは ` ```bash ` ブロックに入れる

---

## 長さの目安

- 1記事あたり Qiita で **5〜8分で読める量**（目安: 800〜1500字 + コード）
- 長くなる場合は記事を分割する
- 「まとめ」は箇条書き5〜7点以内に収める
