RustでTCPチャットサーバーを作る学習記録シリーズです。  
チュートリアルを少しやった程度の初心者が、実際にコードを書きながら理解したことをまとめています。

## なぜRustには所有権があるのか

他の言語にはない「所有権」という概念。最初は「なぜ変数に代入しただけで元の変数が消えるの？」と戸惑います。

これを理解するには、まず**メモリ管理の問題**から入るとスッキリします。

### ダングリングポインタ問題

```rust
// イメージ（実際にはコンパイルエラーになる）
let a = String::from("hello");
let b = a;  // b も同じアドレスを指す（と仮定）

// a のデータを削除
// → b は削除済みのアドレスを指したまま = ダングリングポインタ
```

`b` が無効なアドレスを参照し続ける状態を**ダングリングポインタ**と呼びます。  
C/C++ では実際に起きる深刻なバグで、セキュリティ脆弱性の主要原因の一つです。

### 各言語の解決策

| 言語 | アプローチ | 代償 |
|---|---|---|
| C/C++ | プログラマーが手動管理 | バグの温床 |
| Java/Python | GCが自動管理 | 実行時に予測不能な停止が起きる |
| **Rust** | 所有権は1つ・コンパイラが管理 | 書き方に制約がある |

---

## スタックとヒープ

所有権が管理するのは主に**ヒープメモリ**です。

```
スタック                     ヒープ
┌──────────────────┐        ┌──────────────────┐
│ a:               │        │                  │
│   ptr ───────────┼───────→│  h e l l o       │
│   len: 5         │        │                  │
│   capacity: 5    │        └──────────────────┘
└──────────────────┘
```

- **スタック**: サイズが固定の値を置く。関数終了時に自動で消える
- **ヒープ**: 実行時にサイズが決まる値を置く。返却タイミングを管理する必要がある

`i32` などの整数はスタックに置かれるためコピーが安く、`String` や `Vec` はヒープを使うためムーブセマンティクスが適用されます。

## 所有権のルール

```rust
let a = String::from("hello"); // a が所有者

let b = a;  // 所有権が b に移動（ムーブ）

println!("{}", a); // ← コンパイルエラー: a はもう使えない
println!("{}", b); // ← OK
```

**所有者がスコープを抜けた瞬間、ヒープメモリが解放されます。**  
GC が不要な理由はここにあります。解放タイミングがコード上で確定するからです。

```rust
{
    let a = String::from("hello");
    // ... 処理 ...
} // ← ここで自動的にヒープを解放。free() を書く必要なし
```

## 借用（Borrow）

所有権を移さずに「読むだけ」使いたい場面は多いです。

```rust
fn print_length(s: String) {  // 所有権を奪う
    println!("{}", s.len());
}

fn main() {
    let a = String::from("hello");
    print_length(a);           // 所有権が移る
    println!("{}", a);         // ← エラー！
}
```

`&`（参照）を使うと所有権を渡さずに済みます。

```rust
fn print_length(s: &String) { // 借用するだけ
    println!("{}", s.len());
}

fn main() {
    let a = String::from("hello");
    print_length(&a);          // 借用するだけ
    println!("{}", a);         // ← OK。a はまだ生きている
}
```

### 借用のルール

```
読む借用（&）       → 同時に何個でもOK
書き換える借用（&mut）→ 同時に1個だけ・かつ読む借用と共存不可
```

「同時に書き換え」を禁止するのは**データ競合（data race）**を防ぐためです。  
このルールはコンパイル時に強制されます。実行時ではありません。

---

## まとめ

- 所有権はヒープメモリの安全な管理のための仕組み
- 所有者は常に1つ。スコープを抜けると自動解放
- `&` で所有権を移さずに借用できる
- 借用ルール（特に `&mut` は1つだけ）はデータ競合を防ぐ
- これらはすべてコンパイル時にチェックされる = 実行時ゼロコスト

次の記事では、これらの知識を踏まえてTCPサーバーの基礎を実装します。
